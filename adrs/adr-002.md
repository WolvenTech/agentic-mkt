# ADR-002: Agent Config Colocated in agentic-mkt

## Status

Accepted

## Date

2026-06-22

## Context

The original plan specified a separate private GitHub repository as the runtime source for `agents/*.json` and `skills/*.md`, read by n8n at execution time via fine-grained PAT. That repository does not exist. During design, the stakeholder chose to colocate runtime agent configs in `agentic-mkt/agents/` and have n8n load them from the same GitHub repo via the GitHub node.

Constraints: n8n on `n8n.wolven.com.br` must fetch configs at runtime; skill content originates in the `skill-vault` catalog; proprietary voice/tone skills stay private.

## Decision

Colocate runtime agent configuration in `agentic-mkt/agents/`:

- `agents/{agent_id}.json` — provider, model, skill refs, output schema
- `agents/skills/*.md` — runtime skill markdown (copied/adapted from skill-vault)

The **Call Agent sub-workflow** loads these files from the `agentic-mkt` GitHub repo using the n8n **GitHub node** and a fine-grained PAT scoped to that repository.

**Defer a separate runtime-config repo** until a client project requires runtime config isolation from domain development.

## Alternatives Considered

### Alternative 1: Create a separate runtime-config repo before green run

- **Description**: Bootstrap a new private repo and sync skill-vault content into it.
- **Pros**: Separates runtime mirror from development repo.
- **Cons**: Extra repo setup in a time-constrained session; dual-repo sync not yet built.
- **Why rejected**: Stakeholder chose single-repo colocation for speed.

### Alternative 2: Inline agent config in n8n workflow nodes

- **Description**: Hardcode prompts and skills as n8n static data or Code node constants.
- **Pros**: Fastest green run; no GitHub fetch step.
- **Cons**: Not version-controlled independently; breaks reusable Call Agent harness pattern.
- **Why rejected**: Violates the goal of GitHub-versioned agent definitions (see [ADR-001](adr-001.md)).

### Alternative 3: Sync script to a separate runtime-config repo first

- **Description**: Build git-based sync from skill-vault → a separate runtime repo before n8n integration.
- **Pros**: Automates drift prevention between catalog and runtime.
- **Cons**: Script development competes with the initial rollout; separate repo still required.
- **Why rejected**: Deferred; manual copy into `agentic-mkt/agents/` suffices.

## Consequences

### Positive

- One repo for domain scaffolds, agent configs, and workflow exports.
- GitHub node fetch pattern is identical to a future split-repo approach — easy migration later.
- Agent JSON schema establishes a provider-agnostic contract for future model swaps.

### Negative

- `agentic-mkt` mixes internal planning artifacts with runtime configs (`agents/`).
- Skill drift between skill-vault catalog and `agents/skills/` until sync automation exists.

### Risks

- **Stale runtime skills** — Mitigation: document the copy procedure in `agents/README.md`; add a sync script later.
- **PAT exposure in n8n** — Mitigation: fine-grained PAT with read-only repo scope; rotate on schedule.

## Implementation Notes

- `agents/*.json` includes `provider`, `model`, `skills[]`, and `output_schema` fields.
- n8n GitHub node paths: `agents/{agent_id}.json`, `agents/skills/{skill_name}.md`.
- A future split may move runtime configs into a dedicated repo with a sync script from `agentic-mkt` and skill-vault.
