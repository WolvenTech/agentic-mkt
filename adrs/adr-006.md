# ADR-006: Use Stage-Aware Agent Contracts and Reference Files

## Status

Accepted

## Date

2026-07-01

## Context

The original Call Agent contract was intentionally narrow. `CallAgentInput` contained task fields only, and `AgentOutput` required `deliverable_markdown`, `resumo`, and `autochecagem`. The agent config supported only `skills`; the Call Agent workflow fetched agent JSON and skill markdown from GitHub before assembling the prompt.

The Content Quality Pipeline needs three self-contained stage agents, each with a role skill and one reference/template file. It also needs stage-specific runtime behavior: write the correct Doc page, route blockers to the correct previous human gate, and advance successful stages to the correct next human gate.

## Decision

Extend the agent runtime contract for the staged pipeline.

Agent configs support first-class reference/template files, fetched alongside skills and included in prompt assembly. Stage agent configs declare:

- `id`
- provider/model settings
- `skills`
- `references`
- stage-aware output schema metadata

The runtime stage output contract uses typed fields instead of the legacy draft envelope:

```ts
interface StageAgentOutput {
  stage: "investigate" | "write" | "format";
  artifact_markdown: string;
  resumo: string;
  self_check: string;
  next_gate: "brief review" | "content review" | "final review";
  blocker_question?: string;
}
```

If `blocker_question` is present, n8n posts a `[CQ-BLOCKER]` comment and returns the task to the configured previous human gate. If no blocker is present, n8n replaces the current stage page, posts a `[CQ-AI]` pointer comment, and advances to `next_gate`.

## Alternatives Considered

### Alternative 1: Keep the legacy output envelope

- **Description**: Preserve `deliverable_markdown`, `resumo`, and `autochecagem`; encode stage-specific details inside markdown.
- **Pros**: Minimal Call Agent parser changes; fewer workflow contract changes.
- **Cons**: Forces routing and blocker semantics into untyped text; makes stage behavior harder to validate.
- **Why rejected**: The staged workflow needs explicit machine-readable fields for stage, blocker, page update, and next gate.

### Alternative 2: Embed templates inside role skills

- **Description**: Avoid changing the Call Agent loader by placing reference templates directly in skill markdown.
- **Pros**: Simpler runtime loader; fewer GitHub fetch paths.
- **Cons**: Blurs instruction and artifact-template ownership; makes templates harder to version and tune independently.
- **Why rejected**: Each stage needs one role skill plus one reference/template file, and the runtime should support that contract directly.

### Alternative 3: Treat blockers as parse failures

- **Description**: Make a blocked stage fail validation and route parse failures into blocker handling.
- **Pros**: Avoids another success-path branch.
- **Cons**: Confuses intentional editorial blocking with runtime failure and loses structured blocker content.
- **Why rejected**: Blockers are expected product behavior, not errors.

## Consequences

### Positive

- Stage outputs became deterministic enough for workflow routing and tests.
- Reference templates can evolve independently from role instructions.
- Blocker behavior became explicit and observable.

### Negative

- `AgentConfig`, Call Agent prompt assembly, parsing, and tests needed a contract migration.
- The legacy `linkedin-writer` config and its tests remain scoped to the legacy behavior rather than being migrated (see [`agents/README.md`](../agents/README.md#agent-json-schema) for how the two contracts coexist).
- The n8n Call Agent workflow needed additional GitHub fetch/merge logic for reference files.

### Risks

- Config changes could break all agents if compatibility isn't handled deliberately. Mitigated with fixtures and parser tests for both stage configs and the retained legacy config.
- A malformed `next_gate` could move tasks incorrectly. Mitigated with enum validation in the parser and workflow-side mapping instead of trusting arbitrary model text.

## Implementation Notes

- Workflow-side stage metadata is preferred over model-selected routing where possible. `next_gate` is validated against the invoked stage.
- Model is held constant at `gpt-4.1-mini` across staged agents unless a later requirement changes that.
- `linkedin-format`'s original responsibilities were split: investigative logic moved to `investigative-brief`; channel formatting remained in the trimmed LinkedIn stage skill.
