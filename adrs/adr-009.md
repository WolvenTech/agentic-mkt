# ADR-009: Strict Staged Output and Early Editorial Doc Pointer Persistence

## Status

Accepted

## Date

2026-07-03

## Context

Live n8n and ClickUp review for a test task showed that the staged workflow could create
ClickUp Docs and stage pages, post pointer comments, and advance task statuses, but it did not
write actual artifacts into the Doc pages. Successful executions reached
`PUT Replace Doc Page Content`, while the resulting ClickUp pages still contained only
placeholder content such as `# Brief` and `# Final Draft`.

The failure had two independent causes:

- The deployed/generated Call Agent subworkflow still prompted and parsed the legacy
  `AgentOutput` contract (`deliverable_markdown`, `resumo`, `autochecagem`) even when called
  with staged agents whose config declared `StageAgentOutput` (`stage`, `artifact_markdown`,
  `resumo`, `self_check`, `next_gate`, optional `blocker_question`). The main workflow expected
  `artifact_markdown`, so it could send an empty or undefined content body to the Doc replace
  request while the n8n execution still appeared successful.
- The main workflow created a new ClickUp Doc but never wrote that Doc pointer to the
  `Editorial Doc Url` custom field. The live task's field remained empty after several runs,
  causing later stages and retries to create additional Docs instead of updating the same
  per-task editorial workspace.

The requirement is stricter than a one-off patch: every stage must update the same
artifact workspace, and that workspace must be stored in `Editorial Doc Url` at the beginning of
the first successful run so all later stages reuse it.

## Decision

Use the strict staged contract as the only valid contract for staged agents, and persist the
editorial Doc pointer immediately after Doc creation before any page read/write work continues.

Concretely:

- `build-call-agent.ts` uses the staged-aware parser dispatcher (`parseCallAgentOutputJs`)
  instead of the legacy-only parser (`parseAgentOutputJs`) so the generated Call Agent workflow
  validates staged agent configs against `StageAgentOutput`.
- `assemblePromptJs` produces a required-output example from the actual agent
  `output_schema`. For staged agents, the prompt shows `artifact_markdown`, `self_check`,
  `stage`, and `next_gate`, not legacy `deliverable_markdown`/`autochecagem` placeholders.
- The Marketing Pipeline fails closed if a staged success path reaches the pointer-comment
  or Doc-replace step without a non-empty `artifact_markdown`.
- When `POST Create ClickUp Doc` succeeds, the workflow writes the created Doc ID/URL to
  the `Editorial Doc Url` custom field before `Doc Ready`, then continues with page lookup and
  stage content replacement using that same Doc pointer.
- All stages read and update pages inside the same per-task Doc. Re-running a stage
  replaces only that stage page; moving to another stage creates or updates that stage's page in
  the same Doc.
- Tests cover the generated Call Agent workflow contract, the generated main workflow's
  Doc pointer write, repeated-stage replacement, and multi-stage reuse of one Doc pointer.

## Alternatives Considered

### Alternative 1: Compatibility shim from `deliverable_markdown` to `artifact_markdown`

- **Description**: Let the main workflow accept legacy `deliverable_markdown` when
  `artifact_markdown` is missing, while also adding the missing Doc pointer write.
- **Pros**: Reduces short-term rollout risk if the live Call Agent subworkflow is still old.
- **Cons**: Preserves ambiguity between legacy and staged agents, allows a staged flow to appear
  green while using the wrong contract, and weakens tests that should catch exactly this class of
  failure.
- **Why rejected**: The bug is a contract drift defect. A shim would hide the drift instead of
  forcing the generated workflows and agent configs back into one contract.

### Alternative 2: Main-workflow fallback only

- **Description**: Patch only the Marketing Pipeline to accept whatever the Call Agent returns
  and store the Doc pointer, deferring Call Agent contract cleanup to a later task.
- **Pros**: Smallest immediate workflow diff.
- **Cons**: Leaves the Call Agent subworkflow lying about staged readiness, keeps staged-agent
  prompt examples wrong, and risks future stages continuing to return legacy fields with missing
  `self_check`, `stage`, and validated `next_gate`.
- **Why rejected**: The staged workflow depends on typed stage outputs for routing, blocker
  handling, and page writes. Deferring the Call Agent fix leaves the core integration boundary
  broken.

## Consequences

### Positive

- Empty Doc updates become impossible on the staged success path unless ClickUp itself drops
  valid submitted content.
- One task maps to one editorial Doc, and every stage/retry works against that same Doc pointer.
- The generated Call Agent workflow, staged agent configs, and Marketing Pipeline all enforce
  the same output schema.
- Live proof can assert the actual user-visible artifact content, not just the presence of
  Docs, pages, comments, and statuses.

### Negative

- The fix is larger than a main-workflow-only patch because it touches both generated n8n
  workflows and the tests around their contract.
- Any stale deployed Call Agent workflow must be rebuilt and redeployed together with the main
  workflow; deploying only one side keeps the system broken.

### Risks

- A stale GitHub branch/reference used by the n8n GitHub node can continue serving old agent
  configs even after local fixes. Mitigated by verifying the deployed workflow's GitHub reference
  and the live `Parse Agent Output` code after deployment.
- If ClickUp's URL custom field rejects a bare Doc ID, the pointer write must store a full Doc
  URL while downstream `Use Existing Doc` parses the Doc ID back out. Mitigated with a focused
  test and a live proof step against the real `Editorial Doc Url` field.
- If the Doc pointer write succeeds but later page replacement fails, the task should retain the
  pointer for retry instead of creating a second Doc. Mitigated by ordering pointer persistence
  before page work and by treating retries as "use existing Doc" once the field is populated.

## Implementation Notes

- `parseCallAgentOutputJs` is used in the generated Call Agent workflow; `parseAgentOutputJs` is
  not kept on the production path for staged calls.
- A generated main-workflow node immediately after `Doc Created` updates
  `/task/{task_id}/field/{editorial_doc_url_field_id}` with the created Doc pointer, then passes
  the original `doc_id` forward.
- `Editorial Doc Url` stores a full ClickUp Doc URL (both `doc.clickup.com` and
  `app.clickup.com` forms are recognized); downstream workflow normalizes the stored value to a
  Doc ID before Docs API calls.
- The live-proof runbook checks that one Doc is reused across Investigate, Write, and
  Format, and that page content is non-placeholder after each stage — see the live proof
  evidence under `logs/content-quality-proof/`.
