# ADR-005: Use Local-First Verification with Live Proof as a Follow-Up Task

## Status

Accepted

## Date

2026-07-01

## Context

The project separates fast local checks from live vendor verification. Local tests cover workflow topology, n8n code equivalence, field mapping, parsers, and regression cases such as `.first()` task identity references. Live checks require ClickUp and n8n credentials and can mutate real tasks, Docs, comments, workflow deployments, or executions.

The Content Quality Pipeline changes multiple live integration surfaces: staged ClickUp statuses, Docs API page writes, custom field Doc pointers, pointer comments, blocker returns, and n8n workflow deployment. The implementation must be safe to build locally before running live proof.

## Decision

Define implementation completion for the first slice of work as local verification only. Required local evidence includes:

- unit tests for stage routing, comment filtering, output parsing, Doc pointer handling, and blocker mapping
- workflow topology tests for the three AI ingress statuses
- n8n generated-code equivalence tests
- `pnpm build:workflows:check`
- an updated dry-run/proof script path that can validate staged assumptions without requiring live execution by default

Live ClickUp/n8n proof remains required before production rollout, but it is scheduled as a separate post-implementation task after local behavior is stable.

## Alternatives Considered

### Alternative 1: Require live gated proof before implementation is complete

- **Description**: Treat vendor connectivity, ClickUp writes, n8n deployment, and execution inspection as mandatory completion criteria for the first implementation task.
- **Pros**: Strongest end-to-end confidence.
- **Cons**: Couples code completion to credential availability, live list state, and vendor side effects.
- **Why rejected**: The first implementation should produce an inspectable local change set. Live proof is important but should be scheduled as its own task.

### Alternative 2: Unit tests only

- **Description**: Skip workflow build checks and proof-script updates.
- **Pros**: Fastest local cycle.
- **Cons**: Misses generated n8n JSON drift and integration assumptions.
- **Why rejected**: The project already treats generated workflow checks as part of the trust surface.

## Consequences

### Positive

- Implementation can proceed without blocking on live credentials or mutating production-like ClickUp objects.
- The generated workflow remains inspectable and deterministic before deployment.
- Live proof can focus on vendor behavior instead of basic code correctness.

### Negative

- Local completion does not prove ClickUp/n8n production behavior.
- A follow-up task is mandatory before broader rollout.

### Risks

- The team may mistake local completion for production readiness. Mitigated by documenting the live-proof step explicitly — see [ADR-008](adr-008.md) for how this risk actually materialized and was closed with an exit-code contract.

## Implementation Notes

- Vendor connectivity checks and live ClickUp/n8n execution inspection are kept out of the first implementation's completion gate.
- A dedicated follow-up task covers live staged proof and deployment readiness (see [`LIVE-PROOF-RUNBOOK.md`](../agents/harness/LIVE-PROOF-RUNBOOK.md)).
