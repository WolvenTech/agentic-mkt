# ADR-007: Tag-Based AI Activity Signaling for Staged Columns

## Status

Accepted

## Date

2026-07-02

## Context

The pre-staged `Marketing Pipeline` workflow used a single `Writing` status as a de facto
parking lot: any task in that column was self-evidently "AI is actively working this." The
staged Content Quality Pipeline ([ADR-004](adr-004.md)) replaced that single status with three
AI-owned columns (`investigate`, `write`, `format`) and three human-gate columns
(`brief_review`, `content_review`, `final_review`). This gives finer-grained stage visibility,
but removes the "is AI actually on this right now" signal: a task sitting in `investigate` could
mean the agent is mid-run, or that it silently failed to trigger, or that it's stuck on a
blocker waiting on a human answer. Live-proof testing surfaced this as a real gap — there is no
way to tell these apart from the ClickUp board alone.

Adding a new status per AI column (e.g. `investigate_in_progress`) would multiply the status
matrix and re-introduce exactly the status-proliferation problem ADR-004 moved away from. The
column already encodes *which* stage; what's missing is a cheap, orthogonal signal for
*whether the agent is actively on it or stuck*.

## Decision

Use ClickUp task tags as an orthogonal in-progress/blocked signal layered on top of the
existing AI-column status, scoped only to the three AI columns:

- On stage start (webhook ingress accepted, before calling the Call Agent sub-workflow): add
  tag `agent-working`.
- On a blocker output (`blocker_question` present): swap `agent-working` for `agent-blocked`.
- On stage success and gate advance, or on return to a previous human gate: remove both
  `agent-working` and `agent-blocked` — the task has left AI-column ownership and the column
  status itself is now the source of truth again.

This required new task-tag support in `src/clickup/client.ts` (`addTaskTag`, `removeTaskTag`
wrapping ClickUp's `POST`/`DELETE /task/{task_id}/tag/{tag_name}`), called from the same n8n
code-node points that already write status and pointer comments
(`src/workflows/marketing-pipeline-n8n.ts`).

Tags are not applied to human-gate columns — a task in a gate column is, by definition, waiting
on a human, so no AI-activity signal is needed there.

## Alternatives Considered

### Alternative 1: Add a fourth "in-progress" status per AI stage

- **Description**: Split each AI column into two statuses, e.g. `investigate` and
  `investigate_running`.
- **Pros**: No new primitive (tags) needed; reuses the existing status field.
- **Cons**: Triples the AI-column status count (3 → 9 with blocked variants), multiplies the
  gate-mapping logic in `stages.ts` and the n8n transition code, and re-introduces the same
  status-sprawl problem the staged migration was meant to fix.
- **Why rejected**: Contradicts the leaner-status direction of ADR-004; tags are a cheaper,
  additive signal that doesn't touch the gate state machine.

### Alternative 2: Rely on the task description or a comment for activity signaling

- **Description**: Write "AI working" / "AI blocked" as a comment or description edit instead
  of a tag.
- **Pros**: No new client method needed; comments already exist for pointers.
- **Cons**: Not visible at a glance on the board/list view (ClickUp tags render as colored
  chips on cards; comments do not), and comment history already carries pointer/blocker
  comments — overloading it removes signal rather than adding it.
- **Why rejected**: Board-level visibility was the explicit requirement, which tags provide
  and comments don't.

## Consequences

### Positive

- Restores the "AI is actively on this" signal lost when `Writing` was split into three
  columns, without adding new statuses.
- Tags are visible directly on ClickUp board/list cards, giving at-a-glance triage.
- Fully additive: no change to the existing status state machine or gate mapping.

### Negative

- New ClickUp API surface (task tags) to maintain in `src/clickup/client.ts`.
- Tag writes are additional API calls per stage transition (two per AI stage: set on entry,
  clear on exit), increasing webhook-handler latency slightly.

### Risks

- Tag writes can fail independently of the main stage write (partial failure: status advances
  but tag doesn't clear). Mitigated by treating tag clear as best-effort/non-blocking — log and
  continue rather than failing the stage on a tag-API error, since the tag is a UX aid, not the
  source of truth.
- If a ClickUp workspace doesn't have `agent-working`/`agent-blocked` as pre-existing tags,
  ClickUp auto-creates them on first use — no manual workspace setup required, but naming drift
  (e.g., someone manually renames the tag) would silently break the signal. Mitigated by treating
  tag names as a constant exported alongside the stage definitions.

## Implementation Notes

- `addTaskTag(taskId, tagName)` / `removeTaskTag(taskId, tagName)` live in
  `src/clickup/client.ts`.
- Called from the same n8n code-node functions in `src/workflows/marketing-pipeline-n8n.ts` that
  already handle stage-entry and stage-exit status writes.
- `AGENT_WORKING_TAG` / `AGENT_BLOCKED_TAG` constants live near `src/marketing-pipeline/stages.ts`
  rather than inlining string literals at each call site.
