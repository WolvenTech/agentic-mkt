# ADR-001: V1 Scope — Happy Path with n8n Orchestration

## Status

Superseded by [ADR-004](adr-004.md) (staged Content Quality Pipeline replaced the single-agent
happy path). The idempotency/dedup-deferral decision below is still the source of truth — no
follow-up ADR was written when dedup was later evaluated.

## Date

2026-06-22

## Context

Wolven needs a marketing content pipeline POC that automates first-draft generation from ClickUp tasks while preserving human review. The team is a one-person marketing function; primary success is reusable agentic workflow patterns, not production throughput. Multiple orchestration options exist: custom ClickUp + n8n + Claude, ClickUp native Super Agents, or broader platform bets.

Constraints: ship V1 in 2–3 hours; agent must remain a pure function; n8n owns all ClickUp mutations; marketing lead reviews output manually in `Review` status.

## Decision

Ship **V1 as the happy path only**:

- ClickUp List `Marketing Pipeline` with status flow `Ready to Work → In Progress → Review`
- n8n main workflow triggered on status `Ready to Work`
- Reusable `Call Agent` sub-workflow loading `agents/*.json` and `skills/*.md` from the agent config repo
- Default worker agent: `linkedin-writer` (Claude Sonnet 4.6)
- n8n posts deliverable as ClickUp comment and moves task to `Review`
- Human marketing lead approves or edits before `Approved → Done`

**Defer to Phase 2:** Verifier agent, revision loop (`revision_count` cap), webhook dedup/idempotency, prompt caching optimization, additional agent types.

## Alternatives Considered

### Alternative 1: ClickUp Super Agents (native, no n8n)

- **Description**: Use ClickUp Brain / Super Agents to generate drafts directly inside the workspace.
- **Pros**: Fastest time-to-demo; no webhook infrastructure; single surface for marketing lead.
- **Cons**: Does not produce reusable n8n harness; locks learnings into ClickUp; limited model/tool composability.
- **Why rejected**: POC success metric is cross-project agentic patterns; n8n + GitHub agent repo is the reusable spine Wolven wants to replicate.

### Alternative 2: V1 with Verifier + Dedup included

- **Description**: Ship happy path plus automated quality gate and webhook idempotency on day one.
- **Pros**: More production-ready; catches duplicate runs and off-spec output early.
- **Cons**: Exceeds 2–3h budget; delays learning whether the core harness works; human review in `Review` already covers quality for V1.
- **Why rejected**: Manual review is sufficient verifier for V1; dedup documented as first Phase 2 priority after happy path is green.

### Alternative 3: Platform bet — multi-agent, multi-channel from day one

- **Description**: LinkedIn + blog + ads agents with routing and analytics from launch.
- **Pros**: Higher ceiling; closer to "agentic marketing org."
- **Cons**: Violates quick-win constraint; one-person team cannot absorb maintenance; no validated harness yet.
- **Why rejected**: Prove the pattern on one workflow before expanding scope.

## Consequences

### Positive

- Reversible, same-day POC with clear separation: ClickUp (GUI), n8n (orchestrator), GitHub (agent config), LLM (compute).
- `Call Agent` sub-workflow becomes a reusable API surface for future Wolven projects.
- Manual `Review` gate preserves brand quality without automating judgment prematurely.

### Negative

- Integration fragility (webhooks, HTTPS, ClickUp API) is owned by the team, not ClickUp.
- Duplicate webhook deliveries may cause double runs until dedup ships.

### Risks

- **Silent webhook failures** — Mitigation: verify n8n public HTTPS, active workflow, test with one task; document troubleshooting.
- **Off-brand drafts without Verifier** — Mitigation: human review in `Review`; acceptance criteria custom field guides worker.
- **POC learnings not captured** — Mitigation: document harness contract (I/O envelopes, repo layout, n8n patterns) as explicit V1 deliverable.

## Implementation Notes

- Agent output contract: JSON with `deliverable_markdown`, `resumo`, `autochecagem`.
- n8n never grants ClickUp write access to the agent; all status moves and comments are workflow nodes.
- Phase 2 should add ingress dedup before full Verifier loop.
