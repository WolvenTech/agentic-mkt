# ADR-008: Enforce Exit-Code Contract for Proof and Green-Run Scripts

## Status

Accepted

## Date

2026-07-02

## Context

[ADR-005](adr-005.md) made local-first verification the first completion gate for the staged
Content Quality Pipeline, and named the exact risk that later materialized: "team may mistake
local completion for production readiness." That risk was mitigated only through documentation,
not through the scripts' own exit behavior.

Live-proof review found this played out directly:

- `src/clickup/green-run-validation.ts`'s `buildEvidence()` computes `validation_status` as
  `"passed"` only when the live workflow was actually executed and verified; otherwise `"ready"`
  (preflight-only, nothing executed) or `"blocked"`. But `main()` only mapped `"blocked"` to exit
  code 2 — both `"ready"` and `"passed"` exited 0. Since `GREEN_RUN_EXECUTE` defaults unset, a
  routine `pnpm green-run` never runs the live happy path yet still exited 0, indistinguishable
  at the shell/CI level from an actual pass.
- `scripts/content-quality-proof.ts`'s local-fallback branch never inspected
  `localProof.passed` before returning; a failed local-check row still exited 0. Most individual
  live checks pushed a `"fail"` evidence row but never threw or set `process.exitCode` — only a
  couple of explicit `throw new Error(...)` calls were caught by `main().catch()`.
- `src/clickup/vendor-gate.ts` already got this right: `runGate()` returns distinct 0/1/2 exit
  codes based on real state, with `VENDOR_GATE_STRICT` gating for CI use.

Live-proof review reported success on paper while the actual green run had not completed, with
no actionable message pointing at what to do next.

## Decision

Bring `content-quality-proof.ts` and `green-run-validation.ts` in line with the
`vendor-gate.ts` pattern: exit codes must distinguish "checked and actually passed" from
"checked but incomplete/unverified" from "checked and failed," and any non-zero exit must print
which specific check(s) failed and the concrete next action to resolve them, so a failure is
never silently reported as success.

Concretely:

- `green-run-validation.ts` `main()`: maps `"ready"` (preflight-only, live execution not run) to
  a distinct non-zero exit code from `"passed"` (exit 0) and `"blocked"` (exit 2) — exit 3 for
  "ready but unverified" — and prints which phase(s) were skipped and the command to run them.
- `content-quality-proof.ts`: propagates `localProof.passed` and any `"fail"` evidence row into
  `process.exitCode` in both the local-fallback and live branches, replacing the throw-only error
  path. Prints the failing check IDs (e.g. `A6`, `A13`) and their evidence message on failure,
  mirroring the structured messages already captured in the JSON evidence file so the terminal
  output doesn't require opening the JSON to understand what broke.
- Both scripts adopt the same exit-code convention as `vendor-gate.ts` (0 = fully verified pass,
  non-zero = anything short of that, with distinct codes per failure category) so CI and local
  usage behave consistently across all three proof scripts.

## Alternatives Considered

### Alternative 1: Documentation-only fix — clarify wording of "ready" vs "passed" in output

- **Description**: Keep exit code 0 for both states, just print clearer status text.
- **Pros**: Minimal code change; no risk of breaking existing CI usage that only checks exit
  code 0.
- **Cons**: This is exactly what ADR-005 already tried (documentation) and it didn't prevent the
  reported incident — a human skimming terminal output after a long task run is the failure mode,
  not a missing explanation.
- **Why rejected**: Already tried and already failed to prevent the reported incident; the gap
  is in automatable signal (exit code), not prose.

### Alternative 2: Make the proof scripts always attempt the live run so "ready" can never happen

- **Description**: Remove the preflight-only mode; every invocation either fully executes or
  fails outright.
- **Pros**: Eliminates the ambiguous middle state entirely.
- **Cons**: Local-first verification (ADR-005) exists specifically because live execution
  requires ClickUp/n8n credentials not available in every environment (e.g., CI without live
  creds, or a quick local sanity check). Forcing live execution removes a deliberately designed
  fast/offline path.
- **Why rejected**: Would undo ADR-005's stated trade-off rather than fix the actual defect,
  which is exit-code mapping, not the existence of a preflight-only mode.

## Consequences

### Positive

- A non-zero exit code became a reliable, automatable signal that something needs attention —
  usable in CI and by any wrapper script — closing the exact gap ADR-005 flagged but didn't
  close.
- Failure output tells the operator what failed and how to fix it without requiring them to open
  the JSON evidence file.
- Consistent exit-code convention across all three proof scripts (`vendor-gate.ts` as the
  shared pattern) reduces cognitive load when switching between them.

### Negative

- Any existing automation or muscle-memory that treats "exit 0" as "done, no further action" for
  `pnpm green-run` must be updated to check for the new distinct codes.
- Slightly more verbose terminal output on failure (printing per-check messages).

### Risks

- Changing exit codes is a breaking change for any script/CI step that greps stdout instead of
  checking `$?`, or that only checks `exit code == 0` without distinguishing categories.
  Mitigated by documenting the exit-code contract in [`LIVE-PROOF-RUNBOOK.md`](../agents/harness/LIVE-PROOF-RUNBOOK.md)
  and [`io-contract.md`](../agents/harness/io-contract.md#proof-and-green-run-exit-code-contract).

## Implementation Notes

- `vendor-gate.ts`'s exit-code structure is the reference implementation.
- Failure message format matches what's already captured in evidence JSON entries (check ID +
  message) instead of introducing a second message format.
