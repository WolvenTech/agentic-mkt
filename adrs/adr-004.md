# ADR-004: Replace Single-Agent Marketing Flow with Staged Content Quality Workflow

## Status

Accepted

## Date

2026-07-01

## Context

The existing `Marketing Pipeline` workflow was a single-agent flow with a revision branch: ClickUp status ingress routed `Ready` and `Needs Review`, then both paths converged into one `Execute Call Agent` path and posted the generated draft as a task comment. The Content Quality Pipeline requirements call for three AI stages with human gates: `Investigate`, `Brief Review`, `Write`, `Content Review`, `Format`, and `Final Review`.

A ClickUp Docs proof validated the staged status strings, one list-scoped Doc per task, one page per stage, page replace-in-place, downstream preservation, task comments, blocker comments, and blocker status returns. It also proved that ClickUp task reads do not expose the created Doc ID, so the workflow needs its own canonical Doc pointer.

## Decision

Replace the single-agent list workflow with the staged Content Quality Pipeline workflow for the marketing list.

The production workflow triggers on entry into `investigate`, `write`, and `format`, routes to the corresponding stage, and advances to the next human gate when the stage succeeds:

1. `investigate` -> `brief review`
2. `write` -> `content review`
3. `format` -> `final review`

The workflow stores one ClickUp Doc per task. The Doc contains one page per stage:

1. `Brief`
2. `Argument`
3. `Final Draft`

The workflow stores the canonical Doc URL/ID in a custom field (`Editorial Doc Url`), and also places the link in the task description for human visibility. The custom field is the machine-readable pointer; pointer comments are activity updates, not the source of truth.

## Alternatives Considered

### Alternative 1: Keep the existing Ready/Writing/Approval flow and add staged mode

- **Description**: Keep the single-agent workflow available while adding staged routing as a separate mode or list.
- **Pros**: Lower migration risk; preserves the existing POC path.
- **Cons**: Doubles workflow paths, increases field/status mapping complexity, and weakens the experiment by keeping two control models active.
- **Why rejected**: Staged routing is a replacement, not an addition. The old flow should not remain as a competing production path.

### Alternative 2: Store stage artifacts as sections in one Doc page

- **Description**: Keep one Doc and one page with replaceable markdown sections for all stages.
- **Pros**: One readable page; simpler human navigation.
- **Cons**: Requires fragile section-replacement logic and harder recovery if heading boundaries drift.
- **Why rejected**: The proof already validated page-level replacement and downstream preservation. One page per stage gives cleaner update boundaries.

### Alternative 3: Use task-description links and comments instead of a custom field

- **Description**: Treat the task description and generated comments as the only Doc locator.
- **Pros**: Avoids adding a custom field.
- **Cons**: Not a reliable machine-readable pointer; parsing comments or descriptions for state is brittle.
- **Why rejected**: The workflow needs a stable pointer because ClickUp does not expose attached Docs from `GET /task`.

## Consequences

### Positive

- The runtime workflow matches the product's control surface directly.
- Stage reruns can replace only the relevant Doc page while preserving downstream pages.
- The Doc pointer is explicit and recoverable by automation.

### Negative

- The old single-agent `Ready/Writing/Approval/Needs Review` tests and field mapping had to be migrated (now complete — those statuses no longer exist on the live list).
- The list required a new custom field before production use.
- The staged workflow needed fresh self-echo filtering for AI-driven status advances.

### Risks

- The replacement could disrupt existing tasks still using the old statuses. Mitigated by documenting the status migration and validating generated workflow topology before deployment.
- The Doc custom field could be missing or misconfigured. Mitigated with field-mapping validation and clear failure handling before stage execution.

## Implementation Notes

- `clickup/field-mapping.json` and `clickup/list-schema.md` include staged statuses and the Doc pointer custom field.
- Staged routing is implemented in generated TypeScript workflow builders (`src/workflows/`), then regenerated into workflow JSON.
- Status updates use `.first()` references for task identity to avoid paired-item lineage failures.
- Generated comment prefixes are deterministic: `[CQ-AI]` and `[CQ-BLOCKER]`.
